# ML System Design Doc
## Picnic — Миграция архитектуры ML для классификации обращений в Customer Support — MVP

> Источник кейса: [Picnic Engineering Blog — How we broke customer support language barriers without breaking production](https://blog.picnic.nl/how-picnic-migrated-ml-architectures-without-sacrificing-operational-continuity-271c0e04014a)

---

## 1. Цели и предпосылки

### 1.1. Зачем идём в разработку продукта?

**Бизнес-цель:**
Заменить устаревшую архитектуру классификации обращений в Customer Support (8 отдельных моделей: 4 канала + 2 языка) на единую мультиязычную модель на базе BERT, способную масштабироваться на новые рынки и языки без переобучения с 0

**Почему станет лучше, чем сейчас:**
- Текущая архитектура (logistic regression + два отдельных токенизатора NL/DE) не масштабируется: запуск Франции требовал построения полностью новых моделей
- Слишком высокая доля неклассифицированных обращений в существующей системе
- Поддержка 8 отдельных моделей — значительный операционный overhead
- Нельзя классифицировать обращения на языках, отличных от языка страны (например, экспаты пишут по-английски)
- Проблема холодного старта: для Франции нет достаточного объёма данных для обучения отдельной модели

**Что будем считать успехом итерации:**
- f-beta=2 score новой модели превышает показатель старой как минимум на 10%
- Доля неклассифицированных обращений снижена на ≥ 50% по всем каналам
- Единая модель успешно работает на всех трёх языках (NL, DE, FR) без country-specific дообучения

---

### 1.2. Бизнес-требования и ограничения

**Бизнес-требования:**
- Система классифицирует входящие обращения по 4 каналам: Email, WhatsApp, In-app feedback, Sentiment
- Поддержка языков: нидерландский,немецкий,французский и других (английский от экспатов)
- Маршрутизация обращений к агентам в соответствии с их квалификацией
- Модели развёрнуты на собственной cloud-инфраструктуре Picnic

**Бизнес-ограничения:**
- Нельзя прерывать работу Customer Support во время миграции — operational continuity обязательна
- Данные клиентов конфиденциальны; использование сторонних LLM API недопустимо
- LLM не выбран: подтверждено через внутренний хакатон — для данной задачи он избыточен
- Срок полной миграции: 8 недель

**Ожидания от итерации:**
1. Proof-of-concept на задаче бинарной фильтрации «пустых» сообщений — валидация архитектуры без риска для основной production-системы
2. По результатам PoC — go/no-go решение о полномасштабной миграции на BERT
3. Полная замена 8 старых моделей 4 новыми глобальными (по одной на канал)

**Описание бизнес-процесса пилота:**
Входящее обращение (Email / WhatsApp / In-app) → классификация интента новой моделью → маршрутизация к агенту соответствующей квалификации.

**Критерии успешного пилота:**
- PoC: перехват ≥ 50% «пустых» сообщений с error rate < 4%
- Full migration: f-beta=2 score выше старой архитектуры на ≥ 10%; улучшение по Email-каналу ≥ 85% по доле неклассифицированных

**Возможные пути развития:**
Переход к динамической системе классификации без предопределённого словаря интентов; возможное введение LLM для более сложных сценариев.

---

### 1.3. Что входит в скоуп итерации

**Входит:**
- PoC: fine-tuning `distilbert-base-multilingual-cased` на задаче бинарной классификации «actionable / unactionable»
- Полная миграция: fine-tuning на мультиклассовой задаче классификации интентов (4 модели: Email, WhatsApp, In-app feedback, Sentiment)
- Формирование единого мультиязычного датасета из всех рынков (NL, DE, FR)
- Валидация модели на данных Франции (cold start сценарий)
- Постепенный rollout: сначала один рынок, затем все

**Не входит:**
- Использование LLM (GPT, Llama, Mixtral и др.)
- Feature engineering помимо текста сообщения (проверяется как гипотеза в PoC)
- Динамическая система классификации без предопределённого словаря интентов

**Качество кода и воспроизводимость:**
- Единый обучающий пайплайн для всех каналов и рынков

**Технический долг:**
- Переход к динамической, event-driven системе классификации
- Возможное введение LLM для сложных сценариев Customer Support

---

### 1.4. Предпосылки решения

- Тексты обращений различаются по длине и стилю в зависимости от канала: Email — длинные, формальные; WhatsApp — короткие, неформальные; In-app — 1–2 предложения. Несмотря на это, принята предпосылка, что единый токенизатор DistilBERT захватывает эту вариативность — проверяется в PoC
- **Ключевая гипотеза:** только текст сообщения (без дополнительных derived features) достаточен для классификации. Подтверждение — необходимое условие для full migration
- Аннотация данных производится агентами CS в процессе работы — дополнительных затрат на разметку не требуется
- Единый датасет из всех рынков даёт достаточный baseline для Франции и устраняет cold start problem
- PoC намеренно выбран на другой, более простой задаче (фильтрация неактивных сообщений), чтобы не рисковать основной production-системой при проверке новой архитектуры

## 2. Методология

### 2.1. Постановка задачи

**Задача 1 — PoC (бинарная классификация):**
Фильтрация «пустых» неактивных сообщений (например, «спасибо», «ок»): `actionable` / `unactionable`.

**Задача 2 — Full Migration (мультиклассовая классификация):**
Классификация входящих обращений по предопределённым категориям интентов для маршрутизации к агентам. Четыре отдельные модели: Email, WhatsApp, In-app feedback, Sentiment.

---

### 2.2. Блок-схема решения

**Бейзлайн (текущая архитектура):**
```
[Входящее обращение]
        │ определение языка
        ▼
[Токенизатор NL] или [Токенизатор DE]
        │
        ▼
[Logistic Regression] ← country-specific модель
        │
        ▼
[Маршрутизация к агенту]
Итого: 8 моделей (4 канала × 2 языка)
```

**MVP (новая архитектура):**
```
[Входящее обращение (любой язык)]
        │
        ▼
[distilbert-base-multilingual-cased tokenizer]
        │  единый токенизатор для всех языков
        ▼
[Fine-tuned DistilBERT] ← одна глобальная модель на канал
        │
        ▼
[Маршрутизация к агенту]
Итого: 4 модели (Email, WhatsApp, In-app, Sentiment — мультиязычные)
```

---

### 2.3. Этапы решения задачи

#### Этап 1 — Подготовка данных

**Целевая переменная:**

| Переменная | Описание | Этап |
|---|---|---|
| `is_actionable` | 0/1 — требует ли сообщение действия агента | PoC |
| `intent_class` | Категория интента из предопределённого словаря агентов CS | Full migration |
| `sentiment` | Тональность обращения | Sentiment модель |

**Признаки:**

| Название данных | Есть ли в компании | Ресурс | Качество проверено |
|---|---|---|---|
| Текст обращений (Email) | Да, внутренняя CS платформа | DE/DS | + |
| Текст обращений (WhatsApp) | Да, внутренняя CS платформа | DE/DS | + |
| Текст обращений (In-app) | Да, внутренняя CS платформа | DE/DS | + |
| Метки интентов (разметка агентами) | Да, исторические данные NL, DE | DS/PO | + |
| Данные по Франции | Да, CS платформа FR | DE/DS | — (малый объём) |

**Результат этапа:** единый мультиязычный датасет с разметкой, разделённый на train/val/test.

---

#### Этап 2 — PoC: валидация архитектуры на бинарной задаче

**Цель этапа:** подтвердить три гипотезы до того, как трогать критическую production-систему:
1. Архитектура DistilBERT превосходит бейзлайн (logistic regression)
2. Только текст сообщения достаточен как входной признак
3. Единый мультиязычный датасет даёт лучшее качество, чем раздельные языковые модели

**Формирование выборки (бейзлайн и MVP):**
- Десятки тысяч сообщений, аннотированных агентами CS, из всех рынков (NL, DE, FR) в едином датасете
- Стратификация по языку и рынку для репрезентативности

**Целевая переменная:** `is_actionable` — согласована с CS Operations командой

**Алгоритм:**
`distilbert-base-multilingual-cased` (HuggingFace). Обоснование: явно указан в статье как выбор, балансирующий производительность модели и ресурсные требования; поддерживает 104 языка; данные клиентов остаются внутри инфраструктуры

**Feature engineering:** только сырой текст сообщения без derived features — намеренный эксперимент для проверки гипотезы о достаточности текста

**Гиперпараметры:** конкретные значения в статье не приводятся; подбираются экспериментально.

**Метрики качества:**

| Метрика | Порог | Источник |
|---|---|---|
| Доля перехваченных unactionable сообщений | ≥ 50% | указано в стаье |
| Error rate (пропуск actionable) | ≤ 4% | указано в стаье |
| Precision (unactionable) | — | *используем для контроля ошибок фильтрации* |
| Macro F1 | — | *используем для общей оценки бинарной классификации* |

**Необходимый результат этапа:** подтверждение всех трёх гипотез → go/no-go решение для full migration.

**Риски:**
- Недостаточный объём данных по Франции → использование данных NL и DE для оценки обобщаемости на новый рынок

**Бизнес-проверка:** оценка результатов PoC совместно с CS Operations командой по итогам одной недели в одном рынке

---

#### Этап 3 — Full Migration: мультиклассовая классификация интентов

**Предпосылка для начала этапа:** PoC подтвердил, что архитектура работает и только текст достаточен как признак.

**Формирование выборки:**
- Миллионы строк размеченных обращений по всем каналам и рынкам в едином датасете
- Единый датасет используется для обучения, тестирования и валидации (явно указано в статье)

**Целевая переменная:** `intent_class` — предопределённые категории интентов из словаря агентов CS; согласованы с бизнесом.

**Алгоритм:** `distilbert-base-multilingual-cased`, fine-tuned отдельно для каждого из 4 каналов. Тот же подход, что и в PoC — подтверждён экспериментально.

**Feature engineering:** только текст сообщения — derived features не используются (подтверждено в PoC).

**Гиперпараметры:** конкретные значения в статье не приводятся; подбираются экспериментально аналогично PoC.

**Метрики качества:**

| Метрика | Порог | Источник |
|---|---|---|
| f-beta=2 (macro) | ≥ baseline + 10% | Статья |
| Доля неклассифицированных (все каналы) | ≤ 50% от baseline | Статья |
| Доля неклассифицированных Email | ≤ 15% от baseline | Статья |
| f-beta=2 в разрезе каждого класса интента | — | *для выявления слабых классов* |
| f-beta=2 в разрезе языка / рынка | — | *для контроля качества для Франции* |

**Обоснование f-beta=2:** β=2 означает, что recall весит вдвое больше precision. Пропустить нужный интент дороже, чем неточно классифицировать; особенно критична высокая стоимость полностью неклассифицированных обращений.

**Необходимый результат этапа:** единый глобальный артефакт модели на канал, работающий на всех рынках и языках.

**Риски:**
- Устаревание словаря интентов → требует участия CS-команды при обновлении тегов
- Языковой дрейф при появлении новых рынков → периодическое переобучение

**Бизнес-проверка:** еженедельный мониторинг f-beta=2 и доли неклассифицированных в разрезе рынков и каналов совместно с CS Operations командой в ходе rollout.

---

#### Этап 4 — Rollout и миграция

**Стратегия:** развёртывание в одном рынке на неделю → валидация метрик → поэтапное подключение остальных рынков.

**Срок:** 8 недель — полный суnsetting всех 8 старых моделей и запуск 4 новых глобальных, включая onboarding Франции

---

## 3. Подготовка пилота

### 3.1. Способ оценки пилота

**PoC:** развёртывание в одном рынке на одну неделю → измерение доли перехваченных unactionable сообщений и error rate на реальном трафике.

**Full migration:** последовательный rollout по рынкам с измерением f-beta=2 и доли неклассифицированных обращений после каждого шага.

### 3.2. Что считаем успешным пилотом

| Метрика | Порог |
|---|---|
| PoC: доля перехваченных unactionable сообщений | ≥ 50% |
| PoC: error rate (пропуск actionable) | ≤ 4% |
| Full migration: f-beta=2 vs baseline | ≥ +10% |
| Full migration: улучшение unclassified, все каналы | ≥ 50% |
| Full migration: улучшение unclassified Email | ≥ 85% |
| Франция без cold start | Качество на уровне NL/DE на старте |

### 3.3. Подготовка пилота

PoC развёртывается в одном рынке и не затрагивает основную систему — вычислительный риск минимален. Ожидаемая вычислительная сложность full migration уточняется по результатам PoC перед стартом этапа 3.

## 4. Внедрение (Production)

### 4.1. Архитектура решения

```
[Email / WhatsApp / In-app feedback]
              │  входящее обращение
              ▼
[Fine-tuned DistilBERT]
  4 глобальных модели:
  - Email intent classifier
  - WhatsApp intent classifier
  - In-app intent classifier
  - Sentiment classifier
              │  intent + confidence score
              ▼
[Routing Engine]
  (маршрутизация к агенту по интенту и квалификации)
              │
              ▼
[CS Agent Platform]
```

### 4.2. Описание инфраструктуры и масштабируемости

**Инфраструктура:** собственная cloud-инфраструктура Picnic.

**Плюсы:**
- Данные клиентов не покидают инфраструктуру — соответствие требованиям конфиденциальности
- Стоимость значительно ниже сторонних hosted LLM API
- Единый пайплайн для всех рынков упрощает масштабирование на новые страны

**Минусы:**
- Статичный словарь интентов требует ручного обновления при появлении новых тем

**Почему лучше альтернатив:**
Hosted LLM API (GPT и др.) — отклонён из-за конфиденциальности данных клиентов и избыточности для данной задачи (подтверждено хакатоном). Self-hosted LLM (Llama, Mixtral) — отклонён как «overkill», усложняет объяснение стейкхолдерам и снижает операционную надёжность.

### 4.3. Требования к работе системы

Конкретные SLA в статье не приводятся. Требования уточняются по результатам PoC на реальном трафике.

### 4.4. Безопасность системы

Конкретные меры безопасности системы в статье не описаны.

### 4.5. Безопасность данных

- Обращения клиентов — персональные данные, обрабатываются строго внутри инфраструктуры Picnic
- Использование сторонних LLM API исключено именно по соображениям безопасности данных клиентов

### 4.6. Издержки

В статье отмечено, что решение обходится значительно дешевле стороннего hosted эквивалента. Конкретные цифры не приводятся.

### 4.7. Integration points

В статье описание взаимодействия сервисов на уровне API не приводится. Входящий поток: обращения из Email, WhatsApp, In-app → модель → Routing Engine → CS Agent Platform.

### 4.8. Риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Устаревание словаря интентов | Высокая | Регулярный аудит тегов с CS-командой, плановое переобучение |
| Языковой дрейф при выходе на новые рынки | Средняя | Включение данных нового рынка в следующий цикл обучения |
| Деградация при cold start нового рынка | Средняя | Данные существующих рынков обеспечивают высокий baseline |
| Сбой во время миграции | Низкая | Постепенный rollout по рынкам + возможность rollback к старым моделям |
